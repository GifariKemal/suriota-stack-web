我们的方法

模块化设计，默认即集成。

SURGE 围绕四个核心模块组织，客户可以单独购买或组合购买：Energy Mapping 负责 kWh 与碳核算，Vessel Tracking 负责船队定位与燃油消耗，Water Analytics 服务 KLHK SPARING 与 IPAL 运营，Wastewater Logger 负责被监管的排放报送。每个模块都自带数据模型、仪表板、告警规则与报表模板，但四者共享同一套底层的租户、身份与通知基础设施，同时运行多个模块的客户因此只有一个登录、一个告警收件箱、一个用户管理界面、一份计费关系。

平台特性刻意走 enterprise-grade，而不仅是 dashboard-grade。SURGE 提供带文档的 REST API 与 webhook 系统，便于与客户已有的 ERP、CMMS 与 BI 工具集成。iOS 与 Android 上的原生移动应用，为不坐工作站的操作员推送告警、关键仪表板与确认流程。合作伙伴可按自有品牌进行白标部署。多渠道告警覆盖 SMS、邮件与 WhatsApp，这在印度尼西亚运营中尤其重要，WhatsApp 常常是最可靠的通知渠道。每一次操作都写入按用户、角色与时间戳锁定的不可变审计轨迹。

部署选项匹配客户的数据主权立场。默认是在 AWS Asia Pacific（雅加达）由 SURIOTA 托管运行，正常运行 SLA 为 99.9%。对本地化要求严格的客户，可在自有基础设施上以 Docker 或轻量级 Kubernetes（K3s）栈部署 SURGE。混合方案日益常见：敏感工艺数据保留在本地，非敏感聚合与高管仪表板运行在 SURIOTA 云中，客户既获得可见性，又避免把原始工艺数据推出设施所带来的合规暴露。

集成被刻意设计得不挑剔。任何使用 Modbus RTU 或 TCP、OPC UA、MQTT、REST，甚至只是 CSV 上传的设备，都可以供给 SURGE。我们不要求客户拆除并替换既有的 SCADA、DCS 或 historian, SURGE 从工厂正在运行的任何系统中读取数据，归一化进自身模型，呈现统一视图。既有投资得以保留，新能力则在其上叠加。正是这一架构原则，使 SURGE 对印度尼西亚的工业既有改造场地（brownfield）切实可行，毕竟那里"拆除替换"几乎从不是一个选项。

模块与集成之下，是一套刻意保守的技术栈：时间序列存储用 InfluxDB 与 TimescaleDB，应用层使用 Python（FastAPI）与 Node.js，Web 前端用 React，移动端用 React Native，基础设施即代码采用 Terraform 与 Ansible。每一项选择都把长期可维护性置于潮流之上。平台的客户期望在 2035 年仍运行 SURGE，而不是当年 2026 年正流行的某个框架。
